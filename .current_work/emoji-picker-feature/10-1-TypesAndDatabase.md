# Phase 10: Client — Types & Database Schema for Emoji Actions

## Objective

Define the TypeScript types for emoji action mappings and create the `entity_emoji_actions` database table (migration 22). This table stores per-entity emoji→action definitions including emotion vectors, metabolism placeholders, and substitution text.

## Task 1 — Add emoji action types to `src/types/emoji.ts`

Append the following types to the existing `src/types/emoji.ts` file (created in Phase 2):

```typescript
// ============================================================================
// Emoji Action Types (Phases 10-17)
// ============================================================================

/**
 * The 8 Ekman/Plutchik base emotions.
 * Must match exactly the backend's emotion.Ekman8Names in harmony-link-private/emotion/ekman8.go.
 */
export type Ekman8Emotion =
  | 'joy'
  | 'sadness'
  | 'trust'
  | 'disgust'
  | 'fear'
  | 'anger'
  | 'surprise'
  | 'anticipation';

/**
 * Array of all Ekman8 emotion names for iteration/validation.
 */
export const EKMAN8_EMOTIONS: Ekman8Emotion[] = [
  'joy',
  'sadness',
  'trust',
  'disgust',
  'fear',
  'anger',
  'surprise',
  'anticipation',
];

/**
 * A single emotion effect: an emotion name + a signed intensity delta.
 * Matches the backend's EmotionEffect struct in modules/cognition/base.go.
 * Delta range: -5.0 to +5.0 (client allows wider range than backend's -3.0 to +3.0
 * to account for aggregation before sending).
 */
export interface EmotionEffect {
  emotion: Ekman8Emotion;
  delta: number; // signed, range -5.0 to +5.0
}

/**
 * Metabolism vector placeholder.
 * Will be expanded in a future iteration when the metabolism simulation is built.
 * For now, stores a type (e.g., "eat", "drink", "sleep") and an item description.
 */
export interface MetabolismVector {
  type: string;    // e.g., "eat", "drink", "sleep"
  item: string;    // e.g., "burger", "water", "coffee"
}

/**
 * An emoji action mapping — defines what happens when a specific emoji is sent.
 * Stored per-entity in the entity_emoji_actions table.
 *
 * Effects are combinable: an action can have emotion + metabolism + substitution,
 * or any subset.
 */
export interface EmojiAction {
  /** Unique ID for this action row */
  id: string;

  /** The entity this action belongs to */
  entityId: string;

  /** The native Unicode emoji character (e.g., '🍔') */
  emojiNative: string;

  /** Emotion effect (null if not set) */
  emotionEffect: EmotionEffect | null;

  /** Metabolism vector placeholder (null if not set) */
  metabolismVector: MetabolismVector | null;

  /**
   * User-written substitution text, or null if auto-generated.
   * This text replaces the emoji in the sent message.
   * Example: "*eats a burger hungrily*"
   */
  substitutionText: string | null;

  /**
   * Whether the substitution text was auto-generated from the effects.
   * If true, regenerating effects will overwrite substitutionText.
   * If false (user-written), it's preserved across effect changes.
   */
  autoGenerated: boolean;

  /**
   * Whether this is a pre-seeded default action.
   * Pre-seeded actions can be edited and deleted by the user.
   */
  isDefault: boolean;

  /** Timestamps */
  createdAt: Date;
  updatedAt: Date;
}

/**
 * AdditionalEffects payload sent with an utterance to the backend.
 * Contains the aggregated emotion effects from all mapped emojis in the message.
 */
export interface AdditionalEffects {
  emotionEffects: EmotionEffect[];
  // metabolismEffects will be added here in a future iteration
}

/**
 * Result of resolving all emoji actions in a message text.
 * Produced by EntityEmojiActionService.resolveMessageActions().
 */
export interface ResolvedMessageActions {
  /** The text with all mapped emojis replaced by their substitution text */
  substitutedText: string;

  /** Aggregated emotion effects from all mapped emojis (already clamped per-emotion) */
  effects: AdditionalEffects;

  /** Whether any substitutions were made */
  hasActions: boolean;
}
```

## Task 2 — Create migration 000022

**File:** `src/database/migrations/000022_add_entity_emoji_actions.ts`

```typescript
/**
 * Migration 022: Add entity_emoji_actions table
 *
 * Stores per-entity emoji→action mappings for the emoji behavior system.
 * Each row defines what happens when a specific emoji is sent in chat:
 * - Emotion vector (Ekman8 emotion + signed intensity delta)
 * - Metabolism vector (placeholder for future metabolism simulation)
 * - Substitution text (replaces the emoji in the sent message)
 */

export const migration022 = `
-- Per-entity emoji action mappings
CREATE TABLE IF NOT EXISTS entity_emoji_actions (
  id TEXT PRIMARY KEY NOT NULL,
  entity_id TEXT NOT NULL,
  emoji_native TEXT NOT NULL,
  emotion_effect TEXT,                -- JSON: {"emotion":"joy","delta":2.0} or NULL
  metabolism_vector TEXT,             -- JSON: {"type":"eat","item":"burger"} or NULL
  substitution_text TEXT,            -- User-written or auto-generated RP text, or NULL
  auto_generated INTEGER NOT NULL DEFAULT 0,  -- 1 if substitution was auto-generated
  is_default INTEGER NOT NULL DEFAULT 0,      -- 1 if pre-seeded default action
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (entity_id) REFERENCES entities(id),
  UNIQUE(entity_id, emoji_native)    -- One action per emoji per entity
);

-- Index for fast lookup by entity (the primary query pattern)
CREATE INDEX idx_emoji_actions_entity_id ON entity_emoji_actions(entity_id);
`;
```

**Design decisions:**
- `UNIQUE(entity_id, emoji_native)` — one action per emoji per entity. If the user assigns a new action to the same emoji, it replaces the old one.
- `emotion_effect` and `metabolism_vector` stored as JSON strings — flexible for future expansion without schema changes.
- `substitution_text` can be NULL — if both effects and substitution are NULL, the row shouldn't exist (but we don't enforce this at DB level for flexibility).
- `auto_generated` flag — when the user changes effects, the system can auto-regenerate the substitution text only if the user hasn't manually edited it.
- `is_default` flag — distinguishes pre-seeded defaults from user-created actions. Used by the "Reset to Defaults" feature.
